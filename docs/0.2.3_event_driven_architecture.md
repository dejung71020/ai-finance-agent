## 1. 배경 및 목적

### 1.1 왜 이벤트 기반 구조가 필요한가

AI 금융 비서는 다음과 같은 특성을 가진다:

- **실시간성**: 거래 발생 즉시 AI 분류 → 코칭 개입 → 자동이체 실행이 연쇄적으로 일어남
- **도메인 독립성**: 7개 도메인(users/assets/transactions/automation/investment/coach/planning)이 서로를 직접 호출하지 않아야 함
- **n8n 통합**: 외부 자동화 워크플로우 엔진이 내부 이벤트를 구독해야 함
- **AI 파이프라인**: LLM 분류, 예측 모델, 코칭 엔진이 비동기로 이벤트를 처리해야 함

```
[동기 방식의 문제]
거래생성 → 카테고리분류(LLM, ~2s) → 예산체크 → 코칭메시지(LLM, ~3s) → 자동이체 ...
= 사용자가 5초+ 대기, 하나 실패 시 전체 롤백

[이벤트 방식의 해결]
거래생성 → 즉시 응답(200ms)
         └→ [이벤트 버스] → 카테고리분류 (비동기)
                          → 예산체크 (비동기)
                          → 코칭 트리거 (비동기)
                          → n8n 자동이체 (비동기)
```

### 1.2 설계 원칙

| 원칙                     | 설명                                                     |
| ------------------------ | -------------------------------------------------------- |
| **도메인 이벤트 소유권** | 이벤트는 발행하는 도메인이 소유, 스키마 변경 권한 보유   |
| **At-Least-Once 보장**   | 중복 처리 가능 설계 (Idempotency Key 활용)               |
| **이벤트 불변성**        | 발행된 이벤트는 수정 불가, 보정 이벤트로 상태 변경       |
| **페이로드 최소화**      | 이벤트에 ID만 포함, Consumer가 필요 시 조회 (Thin Event) |
| **순서 보장 범위**       | 동일 user_id 내에서만 순서 보장 (파티셔닝 키)            |

## 2. 기술 선택 (ADR)

> **ADR-001**: 이벤트 스트리밍 기술 선택

### 2.1 후보 비교

| 항목                 |     Redis Streams      |    Apache Kafka    |     RabbitMQ     |
| -------------------- | :--------------------: | :----------------: | :--------------: |
| **현재 인프라 활용** | ✅ (Upstash 기존 연결) |  ❌ (신규 인프라)  | ❌ (신규 인프라) |
| **운영 복잡도**      |          낮음          |        높음        |       중간       |
| **처리량 (TPS)**     |         ~100K          |        ~1M+        |       ~50K       |
| **메시지 보존**      |       설정 가능        |     장기 보존      |     단기 큐      |
| **Consumer 그룹**    |        ✅ 지원         |      ✅ 지원       |     ✅ 지원      |
| **재처리 (Replay)**  |       ✅ ID 기반       |   ✅ Offset 기반   |    ❌ 제한적     |
| **n8n 연동**         |      Webhook 경유      |     Kafka 노드     |    AMQP 노드     |
| **Upstash 지원**     |   ✅ (현재 사용 중)    | ✅ (Upstash Kafka) |        ❌        |
| **개발 속도**        |          빠름          |        느림        |       중간       |

### 2.2 결정

```
Phase 0~4 (현재 ~ 6주차): Redis Streams
Phase 7    (7주차+):       Upstash Kafka로 마이그레이션 (7.5.1)
```

**근거:**

1. Upstash Redis가 이미 연결되어 있어 인프라 추가 없이 즉시 사용 가능
2. 금융 비서 MVP 단계에서 예상 TPS는 100K 이하 (Redis Streams 처리 범위 내)
3. Upstash는 Redis와 Kafka를 동일 계정에서 제공 → 마이그레이션 경로 확보
4. Redis Streams의 XGROUP/XACK 패턴이 Kafka Consumer Group과 개념적으로 동일 → 코드 변경 최소화

### 2.3 마이그레이션 전략 (Phase 7)

```
Redis Streams → Kafka
├── 스트림명 → 토픽명 1:1 매핑
├── XGROUP    → Consumer Group
├── XADD      → Producer.send()
├── XREADGROUP → Consumer.poll()
└── XACK      → Consumer.commit()
```

추상화 레이어(`app/core/event_bus.py`)를 통해 구현체 교체 시 비즈니스 로직 변경 없음.

## 5. 이벤트 스키마 설계

### 5.1 이벤트 Envelope (공통)

모든 이벤트는 동일한 Envelope 구조를 가진다.

```python
# app/core/events/schemas.py

from pydantic import BaseModel, Field
from uuid import UUID, uuid4
from datetime import datetime
from typing import Any, Optional

class EventEnvelope(BaseModel):
    # --- 메타데이터 ---
    event_id: UUID = Field(default_factory=uuid4)       # 중복 처리 방지 키
    event_type: str                                       # "transactions.transaction.created"
    event_version: str = "1.0"                           # 스키마 버전 (하위 호환성)
    occurred_at: datetime = Field(default_factory=datetime.utcnow)

    # --- 라우팅 ---
    source_domain: str                                    # "transactions"
    user_id: Optional[UUID] = None                       # 파티셔닝 키 (순서 보장 단위)
    correlation_id: Optional[UUID] = None                # 요청 추적용 (X-Request-ID)

    # --- 페이로드 ---
    payload: dict[str, Any]                              # 도메인별 데이터

    # --- 재시도 메타 (Consumer가 기록) ---
    retry_count: int = 0
    last_error: Optional[str] = None
```

### 5.3 스키마 버전 관리

```
이벤트 스키마 변경 규칙:

```

## 6. 스트림 / 토픽 설계

### 6.1 Redis Streams 구조

```
finance:users          → users 도메인 이벤트 전용
finance:assets         → assets 도메인 이벤트 전용
finance:transactions   → 거래 이벤트 (최고 빈도, MAXLEN 10만 유지)
finance:automation     → 자동화 실행 이벤트
finance:investment     → 투자 포트폴리오 이벤트
finance:coach          → 코칭/인사이트 이벤트
finance:planning       → 목표/저축 이벤트
finance:dlq            → Dead Letter Queue (재처리 실패 이벤트 보관)
```

### 6.2 스트림별 보존 정책

| 스트림                 | MAXLEN  | 보존 기간 | 근거                                 |
| ---------------------- | ------- | --------- | ------------------------------------ |
| `finance:transactions` | 100,000 | ~30일     | 최고 빈도, 분류 완료 후 DB 영구 저장 |
| `finance:assets`       | 50,000  | ~14일     | 잔액 변동 빈도 중간                  |
| `finance:users`        | 10,000  | ~90일     | 발생 빈도 낮음, 감사 로그 성격       |
| `finance:automation`   | 50,000  | ~30일     | 실행 이력 추적                       |
| `finance:dlq`          | 10,000  | 무제한    | 수동 확인/재처리 필요                |
| 나머지                 | 20,000  | ~14일     | 기본값                               |

### 6.3 XADD 키 구조

```
Stream Key:   finance:{domain}
Message ID:   * (Redis 자동 생성 — 타임스탬프 기반, 순서 보장)
Fields:
  event_id:   {UUID}
  event_type: {string}
  user_id:    {UUID}          ← 파티셔닝 의사결정 기준
  payload:    {JSON string}
```

### 7.2 Consumer 구현 패턴

```python
# app/core/event_bus.py (추상화 레이어)

class EventBus:
    """Redis Streams 기반 이벤트 버스. Phase 7에서 Kafka로 교체 가능."""

    async def publish(self, stream: str, event: EventEnvelope) -> str:
        """이벤트 발행 → XADD finance:{domain} * ..."""
        ...

    async def consume(
        self,
        stream: str,
        group: str,
        consumer_id: str,
        handler: Callable[[EventEnvelope], Awaitable[None]],
        batch_size: int = 10,
    ) -> None:
        """
        XREADGROUP GROUP {group} {consumer_id} COUNT {batch_size} STREAMS {stream} >
        처리 성공 → XACK
        처리 실패 → 재시도 큐 또는 DLQ
        """
        ...

# 도메인 서비스에서 사용 예시
class TransactionService:
    async def create(self, data: TransactionCreate, db: Session) -> Transaction:
        tx = self.repo.create(data)

        await event_bus.publish(
            stream="finance:transactions",
            event=EventEnvelope(
                event_type="transactions.transaction.created",
                source_domain="transactions",
                user_id=tx.user_id,
                payload={"transaction_id": str(tx.id), "amount": tx.amount, ...}
            )
        )
        return tx
```

## 8. 에러 처리 및 재시도 전략

### 8.1 재시도 정책

```
이벤트 처리 실패 시:

1차 실패 → 즉시 재시도 (1회)
2차 실패 → 30초 후 재시도
3차 실패 → 5분 후 재시도
4차 실패 → Dead Letter Queue (finance:dlq) 이동 + 알람 발송
```

### 8.2 Pending Entry 처리

Redis Streams의 PEL(Pending Entry List)에 남은 미처리 이벤트:

```
주기: 매 5분 XPENDING 조회
조건: idle_time > 60초 인 메시지
처리: XCLAIM → 재처리 워커로 소유권 이전
한도: retry_count >= 4 → DLQ 이동
```

### 8.3 멱등성 (Idempotency) 보장

```python
# Consumer는 반드시 event_id 기반 중복 처리 방지
async def handle_transaction_created(event: EventEnvelope):
    idempotency_key = f"processed:{event.event_id}"

    if await redis_client.get(idempotency_key):
        return  # 이미 처리됨, 스킵

    # 비즈니스 로직 실행
    await classify_transaction(event.payload["transaction_id"])

    # 처리 완료 마킹 (TTL 24시간)
    await redis_client.set(idempotency_key, "1", expire=86400)
```

### 8.4 금융 데이터 정합성

```
이벤트 발행 순서:
① DB 트랜잭션 커밋 성공
② 이벤트 발행 (XADD)

← DB 커밋 전 이벤트 발행 금지 (데이터 없는 이벤트 방지)
← 이벤트 발행 실패 시 DB 롤백하지 않음 (DLQ에서 재발행)
← Outbox Pattern 도입은 Phase 7 이후 검토
```
