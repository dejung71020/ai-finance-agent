## 1. 브랜치 전략

### 구조

main ──────────────────────────────────────── (배포 브랜치)
└── feature/{WBS-ID}-{설명}                 (기능 개발)
└── fix/{WBS-ID}-{설명}                     (버그 수정)
└── etc/{설명}                              (설정·문서·비기능)

### 규칙

| 브랜치 | 용도 | 병합 방식 |
|--------|------|-----------|
| `main` | 배포 가능 상태 유지, 직접 push 금지 | PR + CI 통과 후 Merge |
| `feature/*` | WBS 단위 기능 개발 | Squash Merge |
| `fix/*` | 버그 수정 | Squash Merge |
| `etc/*` | 설정·문서·의존성 변경 | Squash Merge |

- `main` 브랜치는 항상 **서버 구동·테스트 통과** 상태를 유지한다.
- 브랜치명에 WBS ID를 포함해 기획 문서와 코드 이력을 추적 가능하게 한다.

## 2. 커밋 컨벤션

### 형식

{scope}: {순번}- {한 줄 설명}

### scope 종류

| scope | 의미 |
|-------|------|
| `{WBS-ID}` | 기능 구현 (예: `1.4.2`, `3.1.1`) |
| `etc` | 설정·문서·의존성·리팩터링 |

### 예시

3.1.1: 1- 거래 카테고리 자동 분류 프롬프트 구현
1.4.2: 1- transaction domain 구현 및 단수복수 형식 맞춤
1.4.2: 2- 오류 수정 후 재커밋
etc: bcrypt 버전 고정 (4.0.1)

### 원칙
- 한 커밋은 **한 가지 논리 단위**만 포함한다.
- 같은 WBS 작업이 여러 커밋일 경우 `순번`으로 구분한다 (`1-`, `2-`).
- 커밋 메시지에 **왜**를 남긴다 (단순 `update` 금지).

## 3. PR (Pull Request) 프로세스

feature 브랜치 push
    → PR 생성 (main 타겟)
    → GitHub Actions CI 자동 실행
        ├── Python 임포트 검사 (서버 구동 가능 여부)
        └── pytest 자동 실행 (tests/ 폴더 존재 시)
    → CI 통과 시 Merge
    → 배포 필요 시 deploy.yml 수동 트리거 (Railway)

### PR 템플릿 체크리스트
- [ ] WBS ID와 브랜치명이 일치한다
- [ ] 새 엔드포인트는 `app/api/router.py`에 등록했다
- [ ] 이벤트 발행은 DB 커밋 성공 후 호출한다
- [ ] `.env` 신규 환경 변수를 GitHub Secrets에도 등록했다
- [ ] CI가 통과한다

## 4. CI/CD 파이프라인

### CI — `ci.yml` (자동, PR 트리거)

PR → main 감지
→ Python 3.11 셋업
→ pip install -r requirements.txt
→ python -c "from app.main import app"   # 임포트 검사
→ pytest tests/ -v                       # tests/ 폴더 있을 때만

**필수 GitHub Secrets**

| Secret | 용도 |
|--------|------|
| `POSTGRESQL_URL` | Supabase DB 연결 |
| `REDIS_URL` | Upstash Redis 연결 (TLS) |
| `SECRET_KEY` | JWT 서명 키 |
| `ENCRYPTION_KEY` | AES-256 암호화 키 |
| `RAILWAY_TOKEN` | Railway 배포 토큰 |

### CD — `deploy.yml` (수동, workflow_dispatch)

- Phase 7 배포 환경 구성 전까지 **수동 트리거**로 운영
- Railway 플랫폼에 Docker 이미지 배포
- `push: main` 자동 트리거는 Phase 7 이후 활성화 예정

## 5. 이슈 · 마일스톤 관리

### 이슈 레이블 체계

| 레이블 | 용도 |
|--------|------|
| `phase-0` ~ `phase-7` | WBS Phase 단위 분류 |
| `feature` | 신규 기능 |
| `bug` | 버그 수정 |
| `infra` | 인프라·설정 |
| `ai` | LLM·Gemini 연동 |

### 마일스톤 → Phase 매핑

| 마일스톤 | 내용 | 기간 |
|----------|------|------|
| Phase 0 | 인프라·인증·이벤트 아키텍처 | 1~2주차 ✅ |
| Phase 1 | 금융 데이터 연동 (자산·거래) | 3~4주차 🔄 |
| Phase 3 | AI 분류·코칭 | 5주차 🔄 |
| Phase 7 | 배포·모니터링 | 10주차 |

## 6. 핵심 결정 사항 (Architecture Decision Records)

| 결정 | 이유 |
|------|------|
| `main` 단일 브랜치 운영 (no develop) | 1인 개발, 릴리즈 주기 없음 → 단순화 |
| Squash Merge | 기능 단위로 히스토리 정리, WBS 추적 용이 |
| CI에 임포트 검사 추가 | 의존성 누락·설정 오류를 PR 단계에서 조기 차단 |
| 수동 배포 (Phase 7 이전) | 실습 중 비용·안전 리스크 최소화 |
| GitHub Secrets 분리 | `.env` 미커밋 원칙, 환경별 값 분리 |
